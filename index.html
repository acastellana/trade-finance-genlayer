SHIPMENT CHECK
Shipment Check
Deterministic settlement handles the normal case. When the importer funds the contract, a 7-day contest window opens — anchored to shipment-check time, not the payment due date. Within that window the importer may contest timing; after it expires, anyone can call acceptShipment() to unblock settlement. Only when shipment timing is contested — and only then — InternetCourt evaluates the one disputed fact and returns a verdict to the same settlement engine. The contract maps that verdict to Release, Refund, or Manual Review.
Shipment accepted by importer. Settlement proceeds to final release.

Architecture

System components

One settlement flow. InternetCourt appears only as a conditional branch at the shipment check step.

A. Normal Settlement — FX Rate
Base → event read by
Relayer → deploys oracle on
GenLayer · 5 validators reach consensus
Relayer → calls receiveRate() on
Base · rate locked on-chain
Trust model: Relayer delivers rate directly. Rate is a public market number — verifiable by anyone. No LayerZero on this path.
B. Disputed Shipment — Court Verdict
Base → dispute event read by
Relayer → deploys court on
GenLayer · 5 AI jurors evaluate evidence
BridgeSender → relayed to
zkSync → forwarded via
LayerZero → delivered to
Base · BridgeReceiver → Factory → Settlement
Trust model: Verdict delivered through LayerZero. Contract only accepts from BridgeReceiver → Factory. Relayer cannot fake a verdict.
